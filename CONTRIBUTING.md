# Contributing to Otter

Thank you for contributing to Otter! Here’s how you can get started:

---

## Commit Message Guidelines

We use [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) as
our commit message format. We also *prefer* the use of [devmoji](https://github.com/folke/devmoji),
but this is just an optional enhancement.

### Why Conventional Commits?

Conventional commits help us:
- Automatically generate changelogs
- Manage semantic versioning
- Maintain a clear project history

### Commit Types

- `feat`: New features (MINOR version bump)
- `fix`: Bug fixes (PATCH version bump)
- `docs`: Documentation changes
- `style`: Code formatting
- `refactor`: Code restructuring
- `test`: Adding/modifying tests
- `chore`: Maintenance tasks
- `perf`: Performance improvements
- `ci`: CI configuration changes
- `build`: Build system changes
- `revert`: Reverts a previous commit

### Message Format

```
<type>[optional scope]: <description>

[optional body]

[optional footer]
```

### Examples

Good commit:
```
feat(renderer): add shader compilation cache

- Improves rendering performance
- Reduces startup time for complex scenes
```

Bad commit:
```
fixed some stuff
```

### Breaking Changes

Use `!` or footer to indicate breaking changes:
```
refactor(scene)!: completely redesign scene management

BREAKING CHANGE: Requires update to scene initialization
```

## Pull Request Process

1. Follow coding standards.
2. Update documentation.
3. PRs will only be merged if they:
   - Pass CI.
   - Are approved by a maintainer.
   - Use Conventional Commit messages.

4. Keep commits clean:
   - If you have many "bad commits" (see above),
     perhaps because you are working on some work-in-progress feature,
     squash them together to give them a proper message.

## General Branching Workflow

Otter follows a structured branching workflow to ensure stability and clear development progress:

- **`main`**: Always contains the latest stable release. No direct commits are allowed here; changes come from merges from `beta`.
- **`beta`**: Serves as the pre-release branch for features in final testing before being considered stable. Merges into `main`.
- **`alpha`**: Used for features that are still under review or require modifications. Merges into `beta` once stable enough.
- **`dev`**: The most dynamic branch where active development and experimentation take place. Changes from `dev` are merged into `alpha` once they are functional and meet initial requirements.

Contributors should base their feature branches off of `dev` and note that branches will only be merged into `dev`, never directly to one of the other branches.

## Adding License Headers

When creating new files (e.g., `.zig`, `build.zig`, or `build.zig.zon`), please include the following license header at the top of the file:
```
// Copyright (c) [YEAR] Otter Engine Contributors
// Licensed under either of:
//   * MIT License (https://opensource.org/licenses/MIT)
//   * Apache License, Version 2.0 (https://www.apache.org/licenses/LICENSE-2.0)
// at your option.
```

Autogenerated files or standard metadata/configuration files (e.g., `.gitignore`, `.editorconfig`) do not need headers.

## Recommended Tools

- Commit hooks: these will validate the commit message, and even emojify them! They will automatically be enabled when you run:
  ```
  npm install
  ```
  in the root of the repo.

- [commitizen](https://commitizen.github.io/cz-cli/): Assists in creating Conventional Commits. Install globally via:
  ```
  npm install -g commitizen
  ```
